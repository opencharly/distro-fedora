# 2026.205.1000 — add check-enc-port-pod — R10 witness for the round-3 volume/port ordering fix

Companion to `opencharly/charly#183` (the deploy-volume-persistence fix). team-lead's
independent verification of that PR's round-2 fix found a BLOCKING regression in
`resolveDeployVolumes`: `config_setup.go`'s port-resolution block runs BEFORE `resolveDeployVolumes` and
writes the per-host overlay entry (`ResolvedPort`) the moment a deploy has ANY
container port, so a PORTED deploy's project-declared `volume:` override was silently
DROPPED on its first `charly config` (the round-2 fix's `!ok`-gated discriminator
treated the port block's overlay write as "already configured").

`check-enc-pod` (this repo's existing R10 witness for the underlying volume-persistence
bug) does NOT exercise this ordering hazard: it reuses the `check-pod` candy, which
declares no `port:`, so its overlay key stays absent until the volume fallback itself
creates it. This entry adds `check-enc-port-pod`, which instead reuses the `web` candy
(`web-layer`, `port: [8080]`, auto-allocated host port — no hardcoded port) specifically
BECAUSE it has a port, and adds its OWN project-declared `type: encrypted` volume
override. Its plan asserts BOTH: the port resolves and is reachable (the port block ran,
exactly as it would for any ported deploy) AND the encrypted volume is initialized and
mounted (the project-declared-volume fallback still fired despite the port block's prior
overlay write) — the live proof for `opencharly/charly`'s
`VolumeProjectChecked`-gated fix.

## Precondition — a pre-seeded enc secret (documented, not a code workaround)

This bed's FIRST live run failed at `[start]` (`charly: encryption passphrase not
available for charly/enc/check-enc-port-pod after 25 attempt(s)`) — RCA'd to a
genuinely pre-existing, unrelated gap: `ResolveEncPassphrase`'s `GOCRYPTFS_PASSWORD`
env-var branch (`sdk/deploykit/enc_passphrase.go:31-33`) never persists the passphrase
to the credential store the way its auto-generate branch does, so a brand-new
encrypted-volume bed's systemd-driven non-interactive re-config (`ExecStartPre`, which
cannot see the interactive shell's env var) can never find it and times out after 2
minutes. `check-enc-pod` never hits this because its secret has been stored on this
dev host since an earlier session; `check-enc-port-pod`, being brand new, had not.
Confirmed via `journalctl --user -u charly-check-enc-pod.service` showing ZERO
"waiting for credential store" lines for the same run window, vs. 24 for
`check-enc-port-pod`'s first attempt.

This does NOT mask this bed's own claim: the project-declared-volume fallback itself
is proven at the earlier `charly config`/mount step (already working before this
precondition was even discovered), fully independent of the credential-store gap.

**Fixed via the sanctioned remediation the tool's own error names** —
`charly secrets set charly/enc check-enc-port-pod <passphrase>` (a one-time bed-setup
step, matching however `check-enc-pod`'s secret was originally established; NOT an R4
code workaround) — documented in this bed's own `description:` so a future clean-host
run diagnoses instantly as a setup precondition, never a code regression.

**Registered for the next thematic batch** (non-blocking per R2 — this bed's own claim
holds without it): persist the env-sourced passphrase too, mirroring the auto-generate
branch's `cred.Write`, closing this reproducibility gap for the WHOLE enc-bed family
(not just this one bed) — high-value, tracked with this RCA as its evidence.
