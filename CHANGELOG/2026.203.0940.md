## What was added

A GROUP bed with a `vm:` HOLDER member (`preempt-vm-holder`, `from: web-vm` ‚Äî
this repo's own lightweight local vm template, already used by
`check-cross-vm-http`; the root superproject's `eval-vm` is a DIFFERENT
project's entity, not resolvable from this repo) declaring `preemptible:
{holds: [vm-test-lock]}`, and a `pod:` CLAIMANT member (`preempt-vm-taker`,
`requires_exclusive: [vm-test-lock]` ‚Äî a token DISTINCT from the existing
pod-only beds' `test-lock`, so this bed never contends with
`check-preempt-arbiter-pod`/`check-preempt-live-pod` over the same synthetic
resource when run concurrently). `bringUpMembers`' deterministic
`sortedMemberKeys` brings the vm holder up first, then the taker, whose own
`charly start` fires `AcquireExclusive` ‚Üí the arbiter finds the RUNNING vm
holder ‚Üí `stopHolders` ‚Üí `pluginHolderStop` gracefully shuts the VM down
(libvirt) and polls until it is actually stopped ‚Üí records a persistent
lease. Two `check:` steps assert the lease surfaces via `charly preempt
status` and that the holder is recorded as preempted.

## Two real bugs this bed caught (both fixed in the paired superproject PR, not here)

Authoring this bed ‚Äî the FIRST disposable bed anywhere in the ecosystem to
exercise the VM-venue arbiter holder path ‚Äî surfaced two pre-existing latent
bugs in the arbiter's VM dispatch, both faithfully ported from the ORIGINAL
`charly/preempt.go` code (never introduced fresh, just never live-tested
before this bed existed):

1. The VM domain name was computed from `addr.Vm` (the shared `vm:` TEMPLATE
   entity, e.g. `web-vm`) instead of the DEPLOY's own name (e.g.
   `preempt-vm-holder`) ‚Äî so the arbiter tried to stop/start a domain that
   never existed under that name. Fixed by keying the domain name off
   `addr.Name` via `vmshared.VmDomainIdentity`, matching the documented rule
   that a bed's libvirt domain is keyed by the DEPLOY name, never the shared
   `kind:vm` entity.
2. The departed-holder existence check for a VM tested whether the
   `domain-state` RPC round-tripped successfully (`ok`), not whether its
   JSON reply actually said the domain exists ‚Äî so a genuinely undefined
   domain (torn down by a sibling teardown step moments earlier) was never
   detected as departed, and the subsequent restore attempt failed hard on
   "domain not found" instead of taking the no-op-departed path. Fixed by
   decoding the reply's `exists` field.

Both are visible live in this bed's log: the FIRST fix is proven by
`bring-up-members` correctly identifying and stopping the running VM holder;
the SECOND is proven by `rebuild-members-down`/`cleanup-members` correctly
taking the "has departed ‚Äî nothing to restore" no-op path with zero
"could not restore" errors, across BOTH the initial teardown and the
fresh-rebuild cycle's re-teardown.
